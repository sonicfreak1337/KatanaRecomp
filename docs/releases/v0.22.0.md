# KatanaRecomp v0.22.0

Release-Datum: 2026-07-14
Status: Pre-Alpha
Runtime-ABI: 5

## Zusammenfassung

v0.22.0 ersetzt den bisherigen flachen Runtime-Speicher durch einen
regionbasierten Dreamcast-Speicherbus. RAM, VRAM, AICA-RAM, BIOS, Flash und
generische MMIO-Geraete besitzen getrennte, testbare Abstraktionen mit
expliziten Aliasfenstern, Zugriffsrechten und Fehlerpfaden.

## Dreamcast-Speicherlayout

- 16 MiB Haupt-RAM mit 28 direkten Aliasfenstern
- 8 MiB VRAM mit linearen 64-Bit- und bankinterleavten 32-Bit-Sichten
- 2 MiB AICA-RAM mit allen direkten Spiegelungen
- read-only 2-MiB-BIOS mit optionalem Abbild
- separates beschreibbares 128-KiB-Flash-Backing
- keine P4-Spiegelung der externen Speicherbereiche
- atomare Ablehnung kollidierender Plattformabbildungen

## MMIO und Speicherdiagnose

- breitenbewusste 8-, 16- und 32-Bit-MMIO-Callbacks
- genau ein Handleraufruf pro breitem Registerzugriff
- standardmaessig strikte natuerliche Ausrichtung
- strukturierte `MemoryAccessError`-Gruende
- globale Zugriffstraces
- adress- und zugriffsgefilterte Watchpoints
- Runtime-ABI 5

## VRAM-Release-Gate

Die 32-Bit-VRAM-Adressumsetzung wird nicht mehr nur mit einzelnen Beispielwerten
geprueft. Das Release-Gate verifiziert alle 8.388.608 moeglichen Byte-Offsets
gegen eine unabhaengig formulierte Bank-/Wort-Berechnung und weist gleichzeitig
nach, dass die Abbildung innerhalb des 8-MiB-Backings bijektiv ist.

Zusaetzliche Bus-Regressionen decken ab:

- alle vier Bytepositionen in beiden VRAM-Baenken
- das letzte Wort vor und das erste Wort nach `0x00400000`
- ausgerichtete 16- und 32-Bit-Zugriffe an Bank- und Wortgrenzen
- mehrere systematische Folgen fortlaufender Woerter in beiden Baenken
- alle 28 interleavten 32-Bit-Aliasfenster an ausgewaehlten Grenzadressen

## Qualitaetssicherung

- 89 automatische Tests
- frischer lokaler Debug-Build mit vollstaendiger Regression
- frischer lokaler Release-Build mit vollstaendiger Regression
- GitHub Actions auf Linux mit GCC
- GitHub Actions auf Windows mit MSVC
- Debug- und Release-Regression auf beiden CI-Plattformen
- der Release-Tag wird erst nach erfolgreichen CI-Checks erstellt

## Rechtliche und projektbezogene Grenzen

- keine kommerziellen ROMs oder Disc-Images
- keine BIOS- oder Flash-Abbilder
- keine urheberrechtlich geschuetzten Assets
- kein uebernommener Code aus Flycast, dcrecomp oder anderen Referenzprojekten
- Referenzprojekte wurden nur zum unabhaengigen Abgleich beobachtbaren
  Adress- und Bankverhaltens verwendet

## Bekannte Einschraenkungen

- unvollstaendiger SH-4-Befehlssatz
- noch keine ausgefuehrte FPU-Semantik
- konkrete Dreamcast-System-, PVR-, AICA-, GD-ROM- und Maple-Register fehlen
- noch keine architekturgerechte SH-4-Ausnahmebehandlung
- nur einfache konstante indirekte Ziele und begrenzte Jump Tables werden
  aufgeloest
